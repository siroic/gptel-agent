* Overview

gptel-agent is an agentic layer built on top of gptel. It provides tool
implementations, sub-agent delegation, agent definition files, and a skill
system. It extends gptel via its preset system — agents ARE presets with
tools and custom system prompts.

** Repository

- Location: =quelpa/build/gptel-agent/=
- Depends on: gptel (=quelpa/build/gptel/=)
- See [[file:../gptel/gptel-ai.org]] for gptel core context

** What gptel-agent Extends

| Component | Extension |
|-----------+-----------|
| Presets | New presets: =gptel-agent=, =gptel-plan=, custom agents |
| Tools | 20+ tools (filesystem, web, remote, introspection, delegation) |
| Tool Preview | Custom overlay-based preview and confirmation UI |
| FSM Handlers | Custom handlers for sub-agent indication (=gptel-agent-request--handlers=) |
| Mode UI | Header-line mode switching (agent ↔ plan) |

** What gptel-agent Does NOT Override

- Core LLM request mechanism (=gptel-request=)
- Backend/model selection
- API parameter handling
- Message encoding/streaming
- Conversation history management

* Architecture

** Indirect Buffer Model

Sub-agents run inside Org indirect buffers whose lifecycle and
relationships are managed exclusively by the canonical API in
=gptel-indirect-buffer.el= (gptel core).  gptel-agent code never calls
=buffer-base-buffer= directly, never walks marker arithmetic by hand,
and never pokes at the IB registry internals — every relationship
query goes through the canonical resolvers.

Lifecycle (all in =gptel-indirect-buffer.el=):

- =gptel-org-ib-insert-child= — the single parent-aware insertion API.
  Inserts a child heading before the parent's =TERMINE= terminator,
  optionally creates an IB narrowed to the new heading.  Returns a
  plist with =:heading-marker= and =:indirect-buffer= keys.
- =gptel-org-ib-create= — open an indirect buffer narrowed to that
  subtree and register a =gptel-org-ib-node= in
  =gptel-org-ib--registry=.
- =gptel-org-ib-close= — tear down the indirect buffer, remove its
  registry node, and leave the parent's structure intact.
- =gptel-org-ib-ensure-terminator= / =gptel-org-ib-streaming-marker= —
  maintain the =TERMINE= sentinel (sole terminator keyword; =RESULTS=
  and =FEEDBACK= no longer play that role).

Resolution (the only sanctioned way to query IB relationships):

- =gptel-org-ib-parent= — parent node or base buffer
- =gptel-org-ib-children= — list of child nodes
- =gptel-org-ib-base= — root base buffer (walks the parent chain)
- =gptel-org-ib-bounds= — =(start . end)= in the parent buffer
- =gptel-org-ib-point= — current point respecting narrowing
- =gptel-org-ib-absolute-position= — =(buffer . pos)= pair

Inconsistencies are fatal: =gptel-org-ib-fatal= signals =user-error=
rather than letting corrupted state propagate.  There are no
=ignore-errors= wrappers around IB operations in gptel-agent.

Design rationale and invariants live in
[[file:../gptel/gptel-indirect-buffer-ai.org][gptel-indirect-buffer-ai.org]]
(see also [[file:../gptel/gptel-ai.org::#ib-canonical-refactor]] for
the refactor history).  This document cross-references them rather
than restating the design.

** Agent Definition Format

Agents are defined in Org files with a =:PROPERTIES:= drawer (preferred)
or Markdown files with YAML frontmatter (legacy).

*** Org Format (current)

#+begin_example
:PROPERTIES:
:name: my-agent
:description: Does specific things
:tools: Read Write Bash
:model: claude-sonnet-4-20250514
:max-tokens: 8192
:backend: Claude
:subagent-models: (:gatherer "Claude/claude-haiku-4-5-20251001" :researcher "Claude/claude-sonnet-4-6")
:include-reasoning: t
:reasoning-effort: high
:pre: (lambda () (require 'gptel-agent-tools-org))
:END:

System prompt text goes here as the body (org markup).

{{AGENTS}} — expanded to list of available agents
{{SKILLS}} — expanded to list of available skills
#+end_example

*** Property Reference

| Property | Org Format | YAML Format | Purpose |
|----------+------------+-------------+---------|
| =name= | =:name: val= | =name: val= | Agent identity |
| =description= | =:description: val= | =description: val= | Brief description (mandatory) |
| =tools= | =:tools: A B C= (space-separated) | YAML list (=- A=) | Available tool names |
| =backend= | =:backend: Claude= | =backend: Claude= | LLM backend |
| =model= | =:model: name= | =model: name= | Model name |
| =max-tokens= | =:max-tokens: 8192= | =max-tokens: 8192= | Token limit |
| =include-reasoning= | =:include-reasoning: t= | =include-reasoning: true= | How to handle LLM thinking blocks (t/nil/ignore) |
| =reasoning-effort= | =:reasoning-effort: xhigh= | =reasoning-effort: xhigh= | Reasoning effort level (low/medium/high/xhigh/max) |
| =subagent-models= | =:subagent-models: (:key "backend/model" ...)= or plist values | YAML nested object | Per-sub-agent model overrides (string or plist) |
| =pre=, =post= | Elisp expression (evaluated) | Elisp expression (evaluated) | Hooks run before/after preset |
| =parents= | Elisp list of symbols (read) | Elisp list (read) | Inherit from other presets |
| =context= | =:context: A B C= (space-separated) | YAML list | Context sources |

Notes:
- In Org format, =:tools= and =:context= are space-separated strings (split automatically)
- =:subagent-models= is an Elisp plist read with =read=, e.g. =(:gatherer "Claude/claude-haiku-4-5-20251001")=
- =:subagent-models= values can be strings or plists: =(:gatherer "Claude/claude-haiku-4-5-20251001" :researcher-deep (:model "Claude/claude-opus-4-7" :reasoning-effort "xhigh" :include-reasoning t))=
- =:include-reasoning= is converted from string: "t" → t, "nil" → nil, "ignore" → 'ignore
- =:reasoning-effort= is kept as string (no conversion needed)
- =:max-tokens= is converted from string to number
- =:pre= and =:post= are Elisp expressions evaluated with =(eval (read val) t)=
- =:parents= is read as Elisp and interned to symbols
- The =CATEGORY= property (added automatically by Org) is ignored

** Two-Mode Operation

1. *Agent Mode* (=gptel-agent= preset) — full tool access, interactive
2. *Plan Mode* (=gptel-plan= preset) — read-only filesystem, planning focus

Switched via header-line buttons in agent buffers.

** Sub-Agent Delegation

- Parent agent calls =Agent= tool with: =type=, =description=, =prompt=
- Sub-agent runs in isolated context (no shared history)
- Task overlay shows progress (Waiting → Calling Tools → Complete)
- Result returned as text to parent agent
- Each sub-agent type has its own agent definition file
- *Handover mode*: when =handover=true=, the sub-agent's output creates a new =AI-DO= heading at the user task level instead of returning text to the caller.  The system navigates up past all agent headings (via =gptel-org-ib-find-user-task-heading=) to find the user-level task heading, then inserts the new =AI-DO= heading as a sibling of the top-level agent subtree.  Parent TODO content (the full user task subtree) is auto-injected into the handover agent's prompt so it can read all accumulated context including triage agent findings.  Used for escalation workflows (triage → specialist).
- *User-initiated handoff*: =gptel-org-handoff= command stops the current task and creates a new =AI-DO= sibling heading for continuation with fresh context.  Useful when a conversation has grown too large for efficient token usage.

** Org Format Instructions

The prompt transform =gptel-org-agent--transform-org-instructions= (in =gptel-org-agent.el=, part of the gptel package) appends org formatting instructions to the system message.  It resolves the heading level dynamically and tells the LLM to respond in org-mode syntax.

Two modes:

- *Legacy (tag-based)*: AI is told to use exact heading levels (e.g. =**** Heading=).  =:assistant:= / =:user:= tags mark conversation turns.
- *TODO keyword mode* (=gptel-org-use-todo-keywords=): AI is told to start with =* AI <title>= using level 1 headings.  A real-time auto-corrector (=gptel-org--auto-correct-stream= on =gptel-post-stream-hook=) rebases heading levels, escapes example blocks, and converts leftover markdown fences as the response streams in.

The legacy tag-based approach is *deprecated* in favor of TODO keyword mode.  See [[file:../gptel/gptel-ai.org::*Discussion Structure: AI/FEEDBACK Keywords]] for full details.

** State Triads

Each agent declares a =:state-words:= triad =(REQUEST DOING DONE)= in
its =:PROPERTIES:= drawer.  The three words are the Org TODO keywords
used for that agent's subtree across its lifecycle:

- *REQUEST* — keyword on the freshly-inserted heading (before the
  sub-agent starts running)
- *DOING* — keyword while the sub-agent is streaming; set by
  =gptel-org-agent= when the sub-agent's indirect buffer opens (IB-5.3)
- *DONE* — keyword after the sub-agent completes; rewritten by
  =gptel-agent-subtree--cleanup= (see [[*Sub-Agent Subtree Handlers]])

Non-agent sub-tasks collapse to three copies of the tool's own
keyword.  When the property is missing the default fallback is
=(PENDING RUNNING DONE)=.

The triad is read via =gptel-agent-state-words= (=gptel-agent.el=) and
consumed from three places:

- =gptel-org-agent.el= — derives the DOING keyword when opening the
  sub-agent's indirect buffer
- =gptel-org.el= — iterates all agent plists and registers every
  triad word into =org-todo-keyword-faces= so they fontify correctly
- =gptel-agent-tools.el= — reads the DONE slot during cleanup-time
  keyword rewriting

** Loading Pipeline

1. =gptel-agent-dirs= specifies directories to scan
2. =gptel-agent--update-agents= discovers =.org= and =.md= files
3. =gptel-agent-read-file= dispatches on extension:
   - =.org= → =gptel-agent-parse-org-properties= (reads =:PROPERTIES:= drawer)
   - =.md= → =gptel-agent-parse-markdown-frontmatter= (reads YAML frontmatter)
4. Template expansion (={{AGENTS}}=, ={{SKILLS}}=)
5. Three top-level agents registered as presets via =gptel-make-preset=:
   =gptel-agent=, =gptel-plan=, =gptel-triage=
6. Tool enum updated for the =Agent= tool's =:enum= (excludes top-level agents)
7. Each agent's =:state-words:= triad (see [[*State Triads]]) is read via
   =gptel-agent-state-words= and its three keywords are registered in
   =org-todo-keyword-faces= (done in =gptel-org.el= at preset-apply time)

** Sub-Agent Subtree Handlers

When =gptel-org-subtree-context= is enabled, sub-agents stream directly
into their own indirect buffer (opened via =gptel-org-ib-create=)
instead of accumulating output into a callback string.  The handler
table =gptel-agent-subtree--handlers= drives a custom DONE chain that
delegates all IB operations to the canonical API in
=gptel-indirect-buffer.el= (see [[*Indirect Buffer Model]]):

| Handler | Purpose |
|---------+---------|
| =gptel-agent-subtree--handle-post-insert= | Like =gptel--handle-post-insert= but omits separator/prefix insertion (sub-agents are non-interactive) |
| =gptel-agent-subtree--handle-done= | Extracts text from the sub-agent's indirect buffer (via =gptel-org-ib-bounds=), then calls parent's =main-cb= |
| =gptel-agent-subtree--cleanup= | Calls =gptel-org-ib-close= to tear down the indirect buffer, rewrites the subtree's TODO keyword from the agent's DOING word to its DONE word (see [[*State Triads]]), advances parent's corrector past the folded subtree |

** Agent Plist Format

#+begin_src elisp
(:name "agent-name"
 :description "Brief description"
 :tools ("tool1" "tool2")         ; list of tool name strings
 :system "System prompt text"
 :backend "Claude"                ; backend name string (resolved to struct on apply)
 :model "model-name"
 :max-tokens 8192
 :include-reasoning t
 :reasoning-effort "xhigh"
 :subagent-models (:gatherer "Claude/claude-haiku-4-5-20251001"
                   :researcher-deep (:model "Claude/claude-opus-4-7"
                                     :reasoning-effort "xhigh"
                                     :include-reasoning t))
 :pre (lambda () ...)
 :post (lambda () ...))
#+end_src

=:subagent-models= maps sub-agent type keywords to model specifications.
Values can be:
- A string like ="Claude/claude-opus-4-7"= (just model override)
- A plist like =(:model "Claude/claude-opus-4-7" :reasoning-effort "xhigh" :include-reasoning t)= (model + extra params)

Extra params from plist values (=:reasoning-effort=, =:include-reasoning=, etc.) are merged into the sub-agent's preset at layer 2 (parent's subagent-models), between the defaults (layer 1) and the agent's own properties (layer 3).

Used by =gptel-agent--get-parent-subagent-model= to override settings when a parent agent spawns sub-agents.

=:agent-handover= is a runtime-only boolean set on the FSM info plist when a
sub-agent is spawned with =handover=true=.  See [[*Tool Call Info Plist (FSM context)]].
