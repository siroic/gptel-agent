#+TITLE: gptel-agent — AI Development Context
#+DESCRIPTION: Reference document for AI-assisted development of gptel-agent

* Overview

gptel-agent is an agentic layer built on top of gptel. It provides tool
implementations, sub-agent delegation, agent definition files, and a skill
system. It extends gptel via its preset system — agents ARE presets with
tools and custom system prompts.

** Repository

- Location: =quelpa/build/gptel-agent/=
- Depends on: gptel (=quelpa/build/gptel/=)
- See [[file:../gptel/gptel-ai.org]] for gptel core context

** What gptel-agent Extends

| Component | Extension |
|-----------+-----------|
| Presets | New presets: =gptel-agent=, =gptel-plan=, custom agents |
| Tools | 20+ tools (filesystem, web, remote, introspection, delegation) |
| Tool Preview | Custom overlay-based preview and confirmation UI |
| FSM Handlers | Custom handlers for sub-agent indication (=gptel-agent-request--handlers=) |
| Mode UI | Header-line mode switching (agent ↔ plan) |

** What gptel-agent Does NOT Override

- Core LLM request mechanism (=gptel-request=)
- Backend/model selection
- API parameter handling
- Message encoding/streaming
- Conversation history management

* Architecture

** Agent = Preset + Tools + System Prompt

#+begin_example
Agent Definition File (Markdown/Org)
  → gptel-agent-read-file (parse frontmatter + body)
  → gptel-make-preset (register as gptel preset)
  → gptel--apply-preset (activate in buffer)
  → LLM receives: system prompt + tool specs + user messages
#+end_example

** Agent Definition Format

Agents are defined in Org files with a =:PROPERTIES:= drawer (preferred)
or Markdown files with YAML frontmatter (legacy).

*** Org Format (current)

#+begin_example
:PROPERTIES:
:name: my-agent
:description: Does specific things
:tools: Read Write Bash
:model: claude-sonnet-4-20250514
:max-tokens: 8192
:backend: Claude
:subagent-models: (:gatherer "Claude/claude-haiku-4-5-20251001" :researcher "Claude/claude-sonnet-4-6")
:pre: (lambda () (require 'gptel-agent-tools-org))
:END:

System prompt text goes here as the body (org markup).

{{AGENTS}} — expanded to list of available agents
{{SKILLS}} — expanded to list of available skills
#+end_example

*** Markdown Format (legacy)

#+begin_example
---
name: my-agent
description: Does specific things
tools:
  - Read
  - Write
  - Bash
model: claude-sonnet-4-20250514
---

System prompt body in markdown.
#+end_example

*** Property Reference

| Property | Org Format | YAML Format | Purpose |
|----------+------------+-------------+---------|
| =name= | =:name: val= | =name: val= | Agent identity |
| =description= | =:description: val= | =description: val= | Brief description (mandatory) |
| =tools= | =:tools: A B C= (space-separated) | YAML list (=- A=) | Available tool names |
| =backend= | =:backend: Claude= | =backend: Claude= | LLM backend |
| =model= | =:model: name= | =model: name= | Model name |
| =max-tokens= | =:max-tokens: 8192= | =max-tokens: 8192= | Token limit |
| =subagent-models= | =:subagent-models: (:key "backend/model" ...)= (plist) | YAML nested object | Per-sub-agent model overrides |
| =pre=, =post= | Elisp expression (evaluated) | Elisp expression (evaluated) | Hooks run before/after preset |
| =parents= | Elisp list of symbols (read) | Elisp list (read) | Inherit from other presets |
| =context= | =:context: A B C= (space-separated) | YAML list | Context sources |

Notes:
- In Org format, =:tools= and =:context= are space-separated strings (split automatically)
- =:subagent-models= is an Elisp plist read with =read=, e.g. =(:gatherer "Claude/claude-haiku-4-5-20251001")=
- =:pre= and =:post= are Elisp expressions evaluated with =(eval (read val) t)=
- =:parents= is read as Elisp and interned to symbols
- The =CATEGORY= property (added automatically by Org) is ignored

** Two-Mode Operation

1. *Agent Mode* (=gptel-agent= preset) — full tool access, interactive
2. *Plan Mode* (=gptel-plan= preset) — read-only filesystem, planning focus

Switched via header-line buttons in agent buffers.

** Sub-Agent Delegation

- Parent agent calls =Agent= tool with: =type=, =description=, =prompt=
- Sub-agent runs in isolated context (no shared history)
- Task overlay shows progress (Waiting → Calling Tools → Complete)
- Result returned as text to parent agent
- Each sub-agent type has its own agent definition file
- *Handover mode*: when =handover=true=, the sub-agent's output creates a new =AI-DO= heading at the user task level instead of returning text to the caller.  The system navigates up past all agent headings (via =gptel-org-ib-find-user-task-heading=) to find the user-level task heading, then inserts the new =AI-DO= heading as a sibling of the top-level agent subtree.  Parent TODO content (the full user task subtree) is auto-injected into the handover agent's prompt so it can read all accumulated context including triage agent findings.  Used for escalation workflows (triage → specialist).
- *User-initiated handoff*: =gptel-org-handoff= command stops the current task and creates a new =AI-DO= sibling heading for continuation with fresh context.  Useful when a conversation has grown too large for efficient token usage.

** TodoWrite Org Integration

When =gptel-org-subtree-context= is enabled in org-mode (automatically set by =gptel-org-mode=):

- =TodoWrite= creates todo headings as *direct children* of the agent heading (no intermediate "Tasks" container)
- Status mapping via =gptel-org-agent-status-keyword-map=: =pending→AI-DO=, =in_progress→AI-DOING=, =completed→AI-DONE=
- When a task transitions to =in_progress=, =gptel-org-agent--redirect-markers-to-heading= moves the FSM's =:position= and =:tracking-marker= into that heading's subtree
- This ensures tool results and AI text stream *under the active task heading*, not as a flat block after all todos
- =gptel-org-agent--collect-todo-headings= uses markers (not positions) to survive keyword-length changes during =org-todo= calls
- The function is idempotent: repeated calls with the same data produce the same structure

Expected org structure:
#+begin_example
,*** AI-DOING My agent task :main@agent:
,**** AI Testing with three tasks
,***** AI-DONE Check current time
,****** ALLOWED Requesting permission to run: Eval
#+begin_src gptel-tool
(Eval :expression "(current-time-string)")
Result: "Sat Apr 11 10:49:58 2026"
#+end_src
,***** AI-DOING Check working directory
(AI work streams here under the active task)
,***** AI-DO Check Emacs version
#+end_example

In non-org buffers, TodoWrite falls back to overlay-based display.

** Org Format Instructions

The prompt transform =gptel-org-agent--transform-org-instructions= (in =gptel-org-agent.el=, part of the gptel package) appends org formatting instructions to the system message.  It resolves the heading level dynamically and tells the LLM to respond in org-mode syntax.

Two modes:

- *Legacy (tag-based)*: AI is told to use exact heading levels (e.g. =**** Heading=).  =:assistant:= / =:user:= tags mark conversation turns.
- *TODO keyword mode* (=gptel-org-use-todo-keywords=): AI is told to start with =* AI <title>= using level 1 headings.  A real-time auto-corrector (=gptel-org--auto-correct-stream= on =gptel-post-stream-hook=) rebases heading levels, escapes example blocks, and converts leftover markdown fences as the response streams in.

The legacy tag-based approach is *deprecated* in favor of TODO keyword mode.  See [[file:../gptel/gptel-ai.org::*Discussion Structure: AI/FEEDBACK Keywords]] for full details.

** Loading Pipeline

1. =gptel-agent-dirs= specifies directories to scan
2. =gptel-agent--update-agents= discovers =.org= and =.md= files
3. =gptel-agent-read-file= dispatches on extension:
   - =.org= → =gptel-agent-parse-org-properties= (reads =:PROPERTIES:= drawer)
   - =.md= → =gptel-agent-parse-markdown-frontmatter= (reads YAML frontmatter)
4. Template expansion (={{AGENTS}}=, ={{SKILLS}}=)
5. Three top-level agents registered as presets via =gptel-make-preset=:
   =gptel-agent=, =gptel-plan=, =gptel-triage=
6. Tool enum updated for the =Agent= tool's =:enum= (excludes top-level agents)

** Preview System

Tools with side effects register preview functions via =gptel--tool-preview-alist=:
- Overlays show proposed action at point
- Syntax-highlighted content (code, diffs)
- Navigation: =n=/=p= keys, toggle with =<tab>=
- Confirm: =RET=, reject: =q=

** Async Tool Pattern

#+begin_src elisp
;; Async tools receive a callback as last argument
(defun my-async-tool (arg1 arg2 callback)
  (do-something-async arg1 arg2
    (lambda (result)
      (funcall callback (format "Result: %s" result)))))
#+end_src

** Sub-Agent Subtree Handlers

When =gptel-org-subtree-context= is enabled, sub-agents use buffer-writing instead of callback-based string accumulation.  The handler table =gptel-agent-subtree--handlers= uses a custom DONE handler chain:

| Handler | Purpose |
|---------+---------|
| =gptel-agent-subtree--handle-post-insert= | Like =gptel--handle-post-insert= but omits separator/prefix insertion (sub-agents are non-interactive) |
| =gptel-agent-subtree--handle-done= | Extracts text from indirect buffer, cleans up, calls parent's =main-cb= |
| =gptel-agent-subtree--cleanup= | Closes indirect buffer, advances parent's corrector past folded subtree |

** Agent Plist Format

#+begin_src elisp
(:name "agent-name"
 :description "Brief description"
 :tools ("tool1" "tool2")         ; list of tool name strings
 :system "System prompt text"
 :backend "Claude"                ; backend name string (resolved to struct on apply)
 :model "model-name"
 :max-tokens 8192
 :subagent-models (:gatherer "Claude/claude-haiku-4-5-20251001"
                   :researcher "Claude/claude-sonnet-4-6")
 :pre (lambda () ...)
 :post (lambda () ...))
#+end_src

=:subagent-models= maps sub-agent type keywords to ="backend/model"= strings.
Used by =gptel-agent--get-parent-subagent-model= to override the default model
when a parent agent spawns sub-agents.

=:agent-handover= is a runtime-only boolean set on the FSM info plist when a
sub-agent is spawned with =handover=true=.  See [[*Tool Call Info Plist (FSM context)]].

** Tool Call Info Plist (FSM context)

#+begin_src elisp
(:position POINT
 :context OVERLAY           ; agent task overlay
 :tool-use [calls...]
 :tracking-marker MARKER
 :error ERROR-DATA
 :transformer FUNC          ; result transformer
 :post (callbacks...)
 :agent-handover BOOL       ; when t, sub-agent uses handover completion flow
 :agent-type STRING         ; sub-agent type name
 :agent-description STRING) ; sub-agent task description
#+end_src

* Error Handling

| Scenario | Mechanism | Recovery |
|----------+-----------+---------|
| File not readable | =error= signal | Caught by gptel, shown as message |
| Tool timeout | =run-at-time= + abort | Callback notifies LLM of failure |
| Process exit non-zero | Exit code check | Error with stderr returned to LLM |
| Malformed frontmatter | =error= in parser | Agent file ignored |
| User abort (=C-c C-c=) | =gptel-abort-hook= | Overlays removed, FSM aborted |
| Missing tool | Excluded from list | Agent works with remaining tools |
